## Documentação Técnica: Agente Ofensivo Autônomo (AOA) - Kali Edition

### I. Visão Geral e Arquitetura do Sistema

O Agente Ofensivo Autônomo (AOA) é um sistema de software projetado para automatizar o planejamento e a execução de cadeias de ataque, adaptando-se às defesas do alvo em tempo real.

O AOA é composto por quatro módulos principais:

| Módulo | Função Principal | Base de Conhecimento |
| :--- | :--- | :--- |
| **1. Base de Conhecimento Multidimensional (B-CM)** | Armazena e permite a busca semântica pelos comandos da tabela `ICE-Breaker.md` (e suas 30 dimensões). | Tabela `ICE-Breaker.md` (como embeddings vetoriais) |
| **2. Core LLM (Decisor Tático)** | O "cérebro" do sistema. Raciocina sobre o estado do alvo e as dimensões da B-CM para selecionar o próximo comando. | LLM (Llama, Mistral ou Qwen ajustado) |
| **3. Orquestrador de Agentes (O-A)** | Gerencia o fluxo de trabalho (Observe, Plan, Act, Adapt) e as condições de falha/sucesso. | Lógica de programação Python |
| **4. Executor de Comandos (EC)** | Interface de execução dos comandos de linha (Kali) e captura de resultados. | Shell do Sistema Operacional (Bash/Zsh) |

### II. Componentes e Pré-Requisitos (Ambiente Kali Linux)

| Categoria | Componente | Propósito |
| :--- | :--- | :--- |
| **Linguagem** | Python 3.x | Linguagem primária para desenvolvimento dos Módulos 1, 3 e 4. |
| **Framework LLM** | `transformers` (Hugging Face) ou Pytorch | Para hospedar e fazer *fine-tuning* do Core LLM. |
| **Base de Dados** | ChromaDB ou FAISS | Banco de dados vetorial para armazenar os *embeddings* da B-CM. |
| **Ferramentas Nativas** | `nmap`, `sqlmap`, `msfconsole`, etc. | Ferramentas pré-instaladas no Kali, acessadas via Módulo 4 (EC). |
| **LLM Base** | Um LLM Open-Source (e.g., Llama, Qwen) | Modelo base que será ajustado para raciocínio tático. |

### III. Passo a Passo de Implementação

#### Fase A: Preparação da Base de Conhecimento Multidimensional (B-CM)

**1. Normalização dos Dados:**

  * Converta a tabela `ICE-Breaker.md` em um formato estruturado (e.g., um DataFrame do Pandas ou um arquivo JSON).
  * **Chave:** A coluna `Comando` é o dado a ser executado. As colunas `D1` a `D29`, `Valência`, `Peso`, e `Arquétipo` são os *metadados* descritivos.

**2. Criação dos Embeddings:**

  * Para cada linha da tabela, concatene o texto das 30 dimensões (D1 a D29, Valência, Peso, Arquétipo) em uma única *string* de descrição.
  * Use um *Encoder* (e.g., um modelo de Sentence Transformer) para converter essa *string* de descrição em um **vetor numérico (embedding)**.

**3. Armazenamento no Banco de Dados Vetorial (B-DV):**

  * Armazene cada vetor no B-DV (ChromaDB/FAISS), mapeando-o ao `Comando` original e seu `ID`.
  * O B-DV permitirá que o Core LLM pesquise por "Comandos onde **Furtividade (D12)** é alta" de forma semântica e eficiente.

#### Fase B: Desenvolvimento do Core LLM (Decisor Tático)

**1. Ajuste Fino (Fine-Tuning) do LLM Base:**

  * Treine o LLM base em um conjunto de dados de "Caminhos de Ataque" que ilustrem o uso das dimensões da B-CM.
      * *Exemplo de Treinamento:* **\[Cenário]** "Preciso de um comando de Reconhecimento Furtivo (Arquétipo) que evite detecção (D12), mas seja robusto (Peso)." **\[Resposta Ideal]** "Pesquisar B-CM por Valência \< 0.5 e Peso \> 0.8. Selecionar ID 2: `hping3...`"
  * Isso ensina o LLM a raciocinar (Chain-of-Thought) usando as dimensões da tabela.

**2. Criação da Função de Busca (RAG):**

  * Desenvolva uma função **Retrieval-Augmented Generation (RAG)** que permite ao Core LLM consultar a B-CM:
    1.  O LLM gera uma *query* semântica (ex.: "Melhores comandos para persistência em ambiente Windows").
    2.  A *query* é vetorizada e usada para pesquisar no B-DV (B-CM).
    3.  Os comandos mais relevantes (e suas dimensões completas) são injetados no contexto do LLM.
    4.  O LLM então escolhe, justifica e formata o comando final.

#### Fase C: Orquestração e Execução (O-A e EC)

**1. Implementação do Executor de Comandos (EC):**

  * Crie uma biblioteca Python (`executor.py`) que utiliza o módulo `subprocess` para executar os comandos do Kali.
    ```python
    import subprocess
    def execute_command(command_str):
        try:
            result = subprocess.run(command_str, shell=True, capture_output=True, text=True, timeout=60)
            return {"output": result.stdout, "error": result.stderr, "exit_code": result.returncode}
        except subprocess.TimeoutExpired:
            return {"output": "", "error": "Timeout", "exit_code": -1}
    ```

**2. Criação do Orquestrador (O-A):**

  * O O-A opera em um ciclo contínuo:

| Etapa | Ação | Uso das Dimensões |
| :--- | :--- | :--- |
| **Observação** | Recebe o `output` do EC e o estado atual do alvo. | N/A |
| **Planejamento** | Envia o estado do alvo e o `output` para o Core LLM. | O LLM usa `D1-Contexto Tático` para decidir o próximo objetivo. |
| **Ação** | O LLM seleciona um comando da B-CM (via RAG) e o envia ao EC. | O LLM usa `Peso` e `Valência` para priorizar o comando mais confiável/apropriado. |
| **Adaptação (Crítica)** | O resultado da execução é *Falha* (`exit_code` \!= 0). | O O-A aciona uma nova pesquisa na B-CM, filtrando por comandos com alta `D2-Adaptação` (Fallback) ou `D27-Resposta` (Auto-adaptação). |

**3. Loop de Aprendizado por Reforço (Avançado):**

  * Implemente uma lógica de *feedback loop*:
      * **Sucesso:** Se um comando for bem-sucedido, o sistema deve aumentar ligeiramente o valor de `Peso` associado ao vetor desse comando na B-CM, e usar esse sucesso como experiência positiva para o treinamento contínuo do LLM.
      * **Falha:** Se um comando falhar de forma recorrente, diminua o `Peso`, incentivando o LLM a explorar comandos com maior `D2-Adaptação`.
